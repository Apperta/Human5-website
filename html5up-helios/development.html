Design Patterns

We have employed the following design patterns into our project:

Model View Controller
Our project has made use of the Model View Controller (MVC) design pattern, allowing us to incorporate consistency across our application. The Data Model handles all the operations associated with database on the server side. Our View is under the templates file, taken charge of the User Interface. Controller plays an intermediate role between View and Model for communication, for each component we are displaying on the client side it holds its own controller.

Singleton
This pattern ensures that only one object of a class is ever created. It is useful when exactly one object is needed to coordinate actions across the system. This design pattern has been used through several parts of the app, we used this for identification credentials when acquiring from Facebook and Google API. The app.js file has restricted it to a single object.

Factory
The Factory design pattern is used across our app, for instance, our graphs in the statistics are generated using a common highCharts framework and a similar dataset structure.

Cards
This is new and has been chosen as one of the design patterns for many companies such as Google, Twitter, Spotify etc. We used this for displaying a big amount of information as introduction at once and only occupy a small section of the screen.

Facade
This Facade design pattern appears in our controller file. The controllers generate all the paths and calls to the database in conjunction with our “group goals” into one single component. It effectively made our code more readable so that it makes it easier to handle the development for the future project.

Refresher
A commonly used design pattern for the entire app, allowing user to pull down the page until it reaches enough distance then release, the whole page will be refreshed.

Testing Strategies


Term 2

Mobile App Compatibility Testing

Mobile App Compatibility Testing focuses on determining if the hardware and software display and allow the application to function properly. It checks and validates that the application behaves as expected across a combination of mobile devices that the users are expected to use to access the application. We verified Mobile App Compatibility on various devices with various screen sizes, resolutions and hardware to ensure Mobile App Compatibility across all devices. In order to carry this out, we made use of emulators/simulators to reduce the need for physical devices for the testing. All throughout the design and implementation process, we made use of Ionic Simulator Testing at each step to make sure that the app meets the design requirements as confirmed with the client. At the same time, we tested the app on various devices like Iphone 5, Iphone 6S, Iphone 6S Plus, Google Pixel, Samsung Galaxy S8 by running the app on Ionic View in each of the devices.

The Compatibility test involved the following steps (processes):
1. Defining the set of environments or platforms the application is expected to work on.
2. Setting-up for testing with different platforms and devices to check that the app runs well under varied configurations.
3. Report bugs. Fix the defects. Re-test to confirm defect fixing.


User Acceptance Testing

User Acceptance testing is a method used to test the product on our end users and check that previously defined requirements are still on track. It also assess the ease of utilisation of an app. The tests take place with real users to measure how ‘usable’ or ‘intuitive’ the app is and how easy it is for users to fulfill their objectives.

Users are asked to perform tasks to see where they encounter problems or experience uncertainty. If multiple people encounter similar problems, suggestions will be made to overcome these usability issues. Essentially, it aims at using a user’s experience to enhance the overall user experience of the app.

We conducted this with the help of our client and few fellow colleagues in the course. Apart from letting them use the app to observe noticeable issues, we also asked them to fill in a short form to gather their feedback on the same to garner some substantial evidence. Unit Testing

Unit testing is a a process in which the smallest teatable parts of an application, called unites, are individually and independdently scrutinized for proper operation. We used Unit Testing to test the functionality of the individual components, like Ratings, Goal Management, Statistics etc. We compared multiple testing frameworks, such as Jasmine, Mocha and QUnit (JavaScript Unit Testing Framework).

In the end we decided to use Jasmine; writing Jasmine tests is clean, simple and elegant. It doesn’t depend on any other JavaScript frameworks i.e no external dependencies.It allows to write tests in the behaviour driven development (BDD) style. Jasmine framework has almost everything built into it (assertions/expectations, test double utilities), bringing the learning curve of a new framework to minimum.

Unit Testing has several advantages that correspond to our project

  • They increase confidence in changing and maintaining code. This is especially useful when developing code concurrently and in iterations, where we have to go back to previously written code to improve it. After every change, we are able to quickly verify that the units still work as intended by running the tests. Furthermore, unit tests can be useful for future developers of the project when they are maintaining the code.
  • It encourages development of more reliable and reusable code because the functions should work according to the goals of the program and should be modular for its unit test to be written.
  • Costs of fixing problems found at this stage is much less than running into them at higher levels
  • Same Unit Tests can be run after every small change, which saves a lot of time since now only the latest changes have to be debugged.[1]


    • Integration Testing

    Integration tests are used to show that the major parts of a system work well together. Integration Testing involves combining different modules and combining them to test as a group. It took place after unit testing. It takes the unit tested modules and applies tests defined in an integration test plan to the aggregates formed from those modules.[2]

    Few methods of Integration Testing:

  • Top Down Integration Testing: The modules at higher levels are tested and integrated first. However, stubs (stubs are modules that act as a replacement for a called module but give the same output as that of the actual product) complicate test management and the delays the testing of the low level units within the cycle.[3]
  • Bottom Up Integration Testing: Each module at lower hierarchy is tested individually at first and then the modules that are built using these modules are tested. It minimises the need for stubs.

    • Once all the features of our system were unit tested, we performed integration testing to test if all major parts of the system work together well. We made use bottom-up integration testing, which involved integrating and testing the lower level units first and then climbing up to the higher level modules.

    Automated Testing

    We are planing to use Mocha and Jasmine, two feature-rich JavaScript test frameworks, to ensure our JavaScript code run smoothly on the applications.
    Mocha is one of the most popular JavaScript testing framework, it is running on both Node.js and in the browser for automatically testing to ensure the high quality of the code.
    Jasmine is a BDD (behavior-driven development) framework which is widely used on testing JavaScript code. It can work on its own and does not rely any other frameworks.

    Accessibility Testing

    To make all users capable for using our applications, there are some situations we need to account for:

  • Different devices have different sreen sizes.
  • People who are blind.
  • People who are colour blind or have trouble on distinguishing low colour contrast.
  • People whose first language is not the language provided by the application want to use our app.

    • Accessibility Tools

    The main accessibility tool seems to be screen readers. A screen reader is a tool used to read what a sighted person will see on the page.
    Unlike on desktops you often need to use a paid for tool like JAWS, luckily on mobile you have free screen readers built-in the operating system.

  • For Android phones:
    • go to Settings > My Device > Accessibility > Vision (new to Android 5.0) > TalkBack > Turn TalkBack on.
  • For iOS phones:
    • go to Settings > General > Accessibility > VoiceOver > Trun VoiceOver on.
